# Instruction: Procurement of New Software (Data Protection & GDPR)

## 1. Purpose

This instruction defines the minimum data protection and GDPR-related requirements when procuring new software that processes personal data.

---

## 2. Core Principle

All software used by the company must comply with GDPR.
Additionally, the company requires that **personal data is hosted and processed within the EU/EEA by default**.

> Note: EU-only data residency is a company policy to reduce risk and complexity. GDPR itself does not strictly require this.

---

## 3. Mandatory Requirements (EU Data Residency Policy)

Before procurement approval, the following must be confirmed:

### 3.1 Data Location

- Personal data must be:

  - stored within the EU/EEA
  - processed within the EU/EEA

### 3.2 Subprocessors

- Vendor must provide a **complete list of subprocessors**
- All subprocessors must be located in the EU/EEA
- Any changes to subprocessors must be communicated

### 3.3 Access Restrictions

- Access to personal data must be limited to EU/EEA
- If access from outside EU exists:

  - it must be documented
  - it must be explicitly approved

---

## 4. Exceptions (Non-EU Transfers)

If EU-only processing is not possible, the following are required:

### 4.1 Legal Transfer Mechanism

One of the following must be in place:

- EU–US Data Privacy Framework (DPF), OR
- Standard Contractual Clauses (SCCs), OR
- other valid GDPR transfer mechanism

### 4.2 Additional Safeguards

- Transfer Impact Assessment (TIA) must be performed (if applicable)
- Risks must be documented and accepted

👉 Exceptions must always be **explicitly approved**

---

## 5. Contractual Requirements

Before using the software, the following must be in place:

- Data Processing Agreement (DPA)
- Defined roles (controller / processor)
- Clear data processing purposes
- Data retention and deletion policy
- Breach notification obligations

---

## 6. Security Requirements

Vendor must demonstrate appropriate security measures, such as:

- ISO 27001, SOC2, or equivalent
- Access control and authentication (e.g. MFA)
- Encryption in transit and at rest
- Incident management process

---

## 7. DPIA Requirement

A Data Protection Impact Assessment (DPIA) must be performed if:

- processing involves sensitive personal data
- processing is large-scale
- individuals are monitored or profiled
- risk to individuals is considered high

---

## 8. Vendor Evaluation Checklist

Before approval, confirm:

- Where is data stored?
- Where is data processed?
- Any transfers outside EU/EEA?
- Transfer mechanism (if applicable)?
- Subprocessor list and locations?
- Any remote access from outside EU?
- Security certifications?
- DPA signed?

---

## 9. Approval

Software may only be approved when:

- all mandatory requirements are met, OR
- documented exceptions are accepted

---

## 10. Summary

- GDPR compliance is mandatory
- EU data residency is the default requirement
- Transfers outside EU must be controlled and justified
- Vendor transparency (especially subprocessors) is critical
